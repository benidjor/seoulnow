## 3. 설계 B — gold compaction (Issue 2) ⭐핵심

### 3.1 plan 의도 추적 — 이것은 "신규"가 아니라 "복원"이다

원래 plan 은 gold compaction 을 처음부터 의도했다.

- `phase-1a-week-2.md:2323`: rewrite placeholder **3개**
  (`rewrite_fact_hotspot_congestion_5min`=gold / `rewrite_dim_place` /
  `rewrite_fact_user_event`)를 SparkSubmitOperator 로 교체.
- spec §189: `iceberg_maintenance` = "**병렬 실행 (rewrite_data_files 3개 동시)** + XCom
  before/after 메트릭 비교".

Day 9 가 의식적으로 silver-only 로 축소했다(결정 A1 = `rewrite_dim_place` 제거, A2 =
task_id 를 `rewrite_silver_hotspot_congestion` 로 rename, gold/dim_place 는 "P1B 후 활성화
의무"). 본 설계는 그 deferred 의도를 **gold 에 한해 복원**한다 (dim_place / user_event 는
여전히 후속).

### 3.2 변경 사항

| 변경 | 내용 |
|---|---|
| 신규 `infra/spark/jobs/compaction_gold.py` | `compaction_silver.py` 미러. `CALL ice.system.rewrite_data_files(table => 'gold.fact_hotspot_congestion_5min', options => map('target-file-size-bytes','134217728'))` + before/after file_count·query_time 출력 |
| 신규 `infra/spark/jobs/expire_snapshots.py` | silver + gold 대상 `CALL ice.system.expire_snapshots`. 3,314 snapshot 축소 → manifest 읽기(`plan_files()` 2.29s) 비용 절감 |
| DAG `rewrite` TaskGroup | `rewrite_gold_fact_hotspot_congestion` child 추가. silver child 와 **병렬** (docstring 이 이미 예고한 구조, `max_active_tis_per_dag=3` 그대로) |
| DAG `expire_snapshots` task | echo placeholder → 실제 `docker run` spark-submit (Day 9 결정 A3 해제). `remove_orphan_files` 는 범위 밖이라 placeholder 유지 |
| `capture_metrics.py` multi-table 일반화 | N개 table 인자 → `{"tables":[{table,files,bytes,snapshots},...]}` 출력. 단일 인자 backward-compat 유지 |
| `callbacks.send_compaction_report` multi-table | before/after 의 table list 를 zip 해 테이블별 reduction 표시. silver(steady-state ≈0 delta) + gold(9,161→N 극적 감소) 둘 다 리포트 |

multi-table 메트릭/리포트는 "3개 동시 rewrite" 설계의 자연스러운 귀결이며, Phase 2 의
dim_place / user_event 추가 시 forward-compatible 하다.

### 3.3 trade-off

- **expire_snapshots = time-travel 이력 손실.** 5분 혼잡도 테이블은 latest 만 쓰므로 무해.
  `older_than` 보수적 설정(예: 0일 이전 전부가 아니라 retain 최소 1~수개)으로 안전장치.
- **rewrite 중 streaming write 충돌 가능성.** DAG schedule 이 `0 3 * * *` (야간, streaming
  peak 회피)라 기존 운영 가정과 동일. Iceberg optimistic concurrency 로 commit 충돌 시 재시도.

## 4. 설계 C1 — endpoint 환경별 분리 (Issue 1)

원인: `.env` 가 `MINIO_ENDPOINT=http://minio:9000`(docker-internal, Day 1 편차) → 호스트
FastAPI 가 AWS C++ SDK(pyiceberg PyArrow S3)로 manifest 읽을 때 `minio` 해석 간헐 실패.
`config.py` 기본값은 이미 `localhost` 다.

plan 의도(`phase-1a-week-2.md:7` "FastAPI Cloudflare Tunnel 노출" + `src/api/main.py`
docstring 의 "/etc/hosts alias" 선결조건)는 **호스트 uvicorn 실행**을 전제한다. Issue 1 은
바로 그 /etc/hosts alias 를 AWS SDK 가 신뢰성 있게 안 쓰는 문제다.

**해결**: 신규 `scripts/run_api.sh` — 호스트 실행 시 `localhost` endpoint 를 env 로 주입.

```bash
#!/usr/bin/env bash
# 호스트(노트북)에서 FastAPI 실행. compose 네트워크 밖이라 minio hostname 대신
# localhost 고정 (Issue 1 — AWS C++ SDK 가 /etc/hosts alias 를 신뢰성 있게 안 씀).
exec env \
  MINIO_ENDPOINT="${MINIO_ENDPOINT:-http://localhost:9000}" \
  LAKEKEEPER_URL="${LAKEKEEPER_URL:-http://localhost:8181}" \
  PYTHONPATH=src \
  uvicorn api.main:app --host 127.0.0.1 --port 8000 "$@"
```

pydantic-settings 우선순위는 **환경변수 > .env** 이므로 wrapper 의 env 가 `.env` 의
`minio:9000` 을 안전하게 override 한다 (정찰에서 5/5 200 검증). docker/VM 내부 실행은 `.env`
의 `minio` 그대로 사용(compose DNS 정상) — 새 컨테이너 추가 없이 환경 분리.

**비범위(후속)**: C2(API 컨테이너화 → compose `minio` 네이티브 해석으로 Issue 1 구조적
제거)는 정찰 doc 이 가리킨 "Oracle VM = compose 네트워크 내부" 형태로, Phase 2 VM 이전 시점의
자연스러운 진화. 지금은 YAGNI.

## 5. 설계 D — FastAPI thread-safety (Issue 3)

원인: `api/deps.py` 의 `@lru_cache` 싱글톤(DuckDB 연결 + pyiceberg Catalog)을 FastAPI 동기
(`def`) 엔드포인트가 anyio threadpool 다중 스레드에서 공유 → 경합/상태 손상 → `Promise.all`
병렬 호출 시 502.

**해결**:

- **DuckDB**: 요청마다 `duck_connection().cursor()` (스레드 독립 cursor). 신규 `duck_cursor()`
  helper 를 `deps.py` 에 추가하고 routes 2곳(`hotspots.py`, `chill_open.py`)이 `duck_connection()`
  대신 `duck_cursor()` 사용. **검증 의무**: DuckDB 임시 SECRET 이 cursor 로 전파되는지 TDD
  동시성 테스트로 확정. 전파 안 되면 cursor 별 S3 config 재주입 또는 connection pool 로 전환.
- **pyiceberg catalog**: `@lru_cache` → `threading.local()` 스레드 로컬 인스턴스.
  `build_catalog()` 0.09s 라 스레드당 1회 생성 비용 무시 가능.

```python
# api/deps.py (설계 골격)
import threading
_tls = threading.local()

def catalog() -> Catalog:
    cat = getattr(_tls, "catalog", None)
    if cat is None:
        cat = build_catalog()
        _tls.catalog = cat
    return cat

def duck_cursor() -> duckdb.DuckDBPyConnection:
    return duck_connection().cursor()  # duck_connection() 은 lru_cache 싱글톤 DB 유지
```

`silver_table_paths` / `gold_table_paths` 는 `_catalog()` → `catalog()` 로 전환.
