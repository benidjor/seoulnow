## 1. silver 0 silent fail — 환경 fix 누적 + root cause 확정 + F2 fallback

**관련 brunch**: `phase-1a/day-3-task-3.4-bronze-to-silver`

### 1.1. 증상

PyFlink Bronze→Silver streaming job 가동 시 bronze 적재는 정상이나 silver 0 row + 디렉토리 자체 미생성.

| 항목 | 결과 |
|---|---|
| smoke 90~150s 동안 bronze 측 적재 | 매 30초 checkpoint 마다 3 row commit, 누적 21+ parquet, MinIO `seoul-warehouse/warehouse/<bronze-uuid>/data/area_code=POIxxx/` 정상 |
| silver 측 row | **0건** |
| silver 측 partition 디렉토리 | **미생성** (silver table 의 `PARTITIONED BY (district)` 의 `district=...` path 0건) |
| stdout / stderr 의 exception / ERROR / WARN | **0건 (silent)** |
| Hadoop CodecPool gzip log | 30초 간격 출력 (bronze checkpoint commit 의 metadata 작성) |

직전 세션 메모리 인계 시점의 가설 3종:
1. UDTF + LATERAL TABLE 의 Flink 1.19 + Iceberg 1.6.1 호환 이슈
2. silver INSERT 의 `FROM ice.bronze.hotspot_raw` 가 batch read 인데 bronze streaming 적재 중 lifecycle 충돌 → silver job 정상 종료 (fail 아님)
3. PyFlink LocalEnvironment mini cluster 의 slot 부족 / deadlock

### 1.2. 직전 세션 (2026-05-06) 환경 fix 7건 누적

| # | 영역 | fix |
|---|---|---|
| 1 | Hadoop classpath | `hadoop-client-api/runtime 3.3.4` (Apache 의 shaded uber JAR). `hadoop-aws` / `aws-java-sdk-bundle` (v1) 제거 |
| 2 | Iceberg AWS SDK | `iceberg-aws-bundle 1.6.1` (Iceberg 의 AWS SDK v2 shaded bundle) 추가 |
| 3 | Lakekeeper credentials vending | Iceberg 1.7.1 → **1.6.1** + Flink 1.20 → **1.19.2** 다운그레이드. 1.7 의 `LoadCredentialsResponseParser` strict validation 회피 (1.6 은 vended credentials parse 자체 안 함) |
| 4 | Flink SQL parser | `CREATE CATALOG IF NOT EXISTS` 1.19 미지원 (FLINK-32777 가 1.20 에 추가). `list_catalogs()` 로 conditional 분기 |
| 5 | LATERAL TABLE syntax | scalar `@udf` → `@udtf` 변경. `CROSS JOIN LATERAL TABLE(SELECT ...)` 문법 거부 → `CROSS JOIN LATERAL TABLE(enrich_hotspot(...))` 정상 syntax |
| 6 | StatementSet | 두 INSERT (bronze + silver) 를 `t_env.create_statement_set()` 으로 atomically submit |
| 7 | MinIO endpoint resolve | `/etc/hosts` 에 `127.0.0.1 minio` 추가. **root cause** — Lakekeeper REST 가 vending 한 endpoint `http://minio:9000/` 를 호스트의 Iceberg client 가 resolve 못 해 `UnknownHostException`. host 의 hosts 매핑으로 해결 |

직전 세션 결과 — bronze 정상화는 되었으나 silver 0 silent fail 그대로. fix #3 의 다운그레이드가 이번 세션 root cause 식별의 결정적 단서가 됨 (1.6 의 silent vs 1.7 의 strict validation).

### 1.3. 본 세션 정밀 진단 + fix #1~5

#### 1.3.5. fix #4 — `header.X-Iceberg-Access-Delegation = remote-signing`

결과 — **같은 stack trace 정확히**. fix 위치가 client side 가 아니라 **Lakekeeper side 로 완전히 확정**.

### 1.4. root cause 확정

> **Lakekeeper v0.5 가 LoadTable REST response 의 `storage-credentials` 필드에 빈 config dict (`[{"prefix": "...", "config": {}}]`) 를 일관되게 포함. Iceberg 1.7.x 의 `Credential.validate()` 가 `checkArgument(!config.isEmpty())` strict validation 으로 즉시 throw.**

근거 사슬:
- Lakekeeper 가 client 의 `X-Iceberg-Access-Delegation` header 의 모든 값 (없음 / `remote-signing` / `vended-credentials`) 에 대해 동일하게 빈 config 응답 — header 무시
- Iceberg 1.6.x 는 이 validation 을 안 해서 빈 config 를 받아들고 그 후 commit 단계에서 silently fail (stack trace 안 던짐)
- Iceberg 1.7.x 는 strict validation → `Invalid config: must be non-empty` IllegalArgumentException 즉시 throw

bronze 가 정상 commit 됐던 이유:
- bronze table 의 첫 commit 은 직전 세션 (2026-05-06) 의 다른 시점에 시작
- 이미 bronze table 의 storage path 가 만들어진 상태 (권한 OK)
- 이후 commit 은 같은 path 재사용 → 새 권한 negotiation 불필요
- silver table 은 이번 세션이 첫 commit 시도 = 새 path 생성 단계에서 storage-credentials negotiation 시도 → 빈 config 거부

## 2. systematic-debugging Phase 4.5 도달 — 12 fix 시도의 임계점

### 2.1. 적용된 권고

systematic-debugging skill 의 Phase 4.5 권고:

> **If 3+ Fixes Failed: Question Architecture**
>
> Pattern indicating architectural problem:
> - Each fix reveals new shared state/coupling/problem in different place
> - Fixes require "massive refactoring" to implement
> - Each fix creates new symptoms elsewhere
>
> STOP and question fundamentals:
> - Is this pattern fundamentally sound?
> - Should we refactor architecture vs. continue fixing symptoms?
>
> Discuss with your human partner before attempting more fixes.

본 세션의 적용:

### 2.2. 본 세션의 학습

- **Phase 4.5 의 1차 적용 (fix #2 실패 후 진단 진입) 이 root cause 식별의 결정적 분기점** — 그 시점에 fix attempt 만 누적했으면 더 깊은 미궁
- **2차 적용 (fix #5 실패 후 fallback 결정) 도 정당** — fix 의 새 영역 (Lakekeeper 업그레이드 / catalog 변경) 은 본 brunch 의 scope 를 벗어나는 큰 변경. F2 fallback 으로 본 brunch 마무리 + Day 4 의 명확한 출발점 확보가 효율적
- **fix attempt 누적 시간의 trade-off** — 본 세션의 5시간+ 가 "낭비" 가 아닌 "root cause 정확한 식별" 의 비용. 단 그 시점이 임계점을 넘었다면 fallback 진입의 명확한 신호
