## 1. Issue 1 — pyiceberg PyArrow S3 의 `minio` hostname 간헐 DNS 해석 실패

### 증상

로컬 FastAPI `/api/hotspots` 를 반복 호출하면 **첫 요청만 200, 이후 전부 500**:

```
hotspots #1: 5.86s HTTP 200
hotspots #2: 5.50s HTTP 500
hotspots #3: 1.49s HTTP 500
```

`/health` 직접 호출은 항상 정상이라 프로세스는 살아있음. uvicorn 기본 stdout 버퍼링 때문에 처음엔 traceback 이 안 보였고, `PYTHONUNBUFFERED=1` 재기동 + 포트 점유 좀비 프로세스 정리 후에야 실제 에러를 포착:

```
OSError: When reading information for key 'warehouse/.../snap-...avro'
in bucket 'seoul-warehouse':
AWS Error NETWORK_CONNECTION during HeadObject operation:
curlCode: 6, Couldn't resolve host name
```

### 원인

`.env` 의 `MINIO_ENDPOINT=http://minio:9000` (docker-internal endpoint, Day 1 환경 편차 SoT). docker 안에서는 `minio` 가 compose DNS 로 풀리지만, **호스트에서 띄운 FastAPI** 는 `/etc/hosts` 의 `127.0.0.1 minio` alias 에 의존한다.

DuckDB 쪽(`configure_duckdb`)은 `minio_endpoint` 를 S3 SECRET 으로 설정해 잘 동작했으나, **pyiceberg 의 `plan_files()` 가 쓰는 PyArrow S3FileSystem(AWS C++ SDK)** 은 manifest(.avro) 를 읽을 때 `minio` 해석을 **간헐적으로 실패**한다. AWS SDK 의 DNS 해석이 `/etc/hosts` 를 일관되게 참조하지 않거나, FastAPI threadpool 스레드에서 negative 캐시/race 가 발생하는 것으로 추정.

### 해결 (정찰 시점 임시)

FastAPI 기동 시 endpoint 를 `localhost`(special-cased, 항상 resolve)로 override:

```bash
MINIO_ENDPOINT=http://localhost:9000 LAKEKEEPER_URL=http://localhost:8181 \
  PYTHONPATH=src uvicorn api.main:app --host 127.0.0.1 --port 8000
```

→ 5회 연속 모두 200 (500 사라짐). DNS 실패가 endpoint hostname 차이의 증상임이 확정.

### Task 11.1 이월 메모

- gold serving 을 호스트 FastAPI 로 운영한다면 endpoint 를 `localhost`/`127.0.0.1` 로 고정하거나, pyiceberg FileIO 의 S3 설정을 명시 주입해야 함.
- Oracle VM 배포 시에는 compose 네트워크 내부라 `minio` 해석이 정상일 가능성 — 환경별 endpoint 분리 설정 필요.

## 2. Issue 2 — gold 테이블 small-file accumulation 으로 인한 read latency (핵심)

### 증상

Issue 1 해결(localhost endpoint) 후 500 은 사라졌으나 응답 시간이 **5s ~ 29s 로 극심하게 변동**:

```
hotspots #1: 6.17s 200   #2: 28.94s 200   #3: 5.05s 200
hotspots #4: 26.20s 200   #5: 5.02s 200
```

4초 edge timeout(Edge API AbortController)으론 절대 못 버티는 수준.

### 원인 (실측)

단계별 분해:

| 단계 | cold | warm |
|---|---|---|
| `build_catalog()` | 0.09s | — |
| `load_table()` | 0.43s | — |
| `scan().plan_files()` (manifest 읽기) | 2.29s | 0.25s |
| DuckDB `read_parquet` (data) | 2.78s | — |

스모킹건:

```
data file 개수: 9,161
snapshot 개수:  3,314
총 row 수:      9,929   → 파일당 약 1 row
```

streaming 이 5분 micro-batch 로 **tiny parquet 9천 개**를 쌓은 전형적 small-file 문제. 병목은 파일 수에 정비례하는 두 곳 — pyiceberg manifest list 읽기 + DuckDB 의 9,161개 파일 S3 read. 로컬 측정은 ~5s 인데 HTTP 가 29s 까지 튀는 것은, 9천 파일을 S3 로 읽을 때 connection 재시도 + (Issue 3 의) 스레드 경합이 겹친 결과.

### 해결 방향 (Task 11.1 이월)

- `rewrite_data_files` (compaction) 으로 9,161 작은 파일 → 소수 큰 파일. plan_files + read 둘 다 파일 수 비례라 **<0.5s 기대**.
- `expire_snapshots` 로 3,314 snapshot metadata 축소.
- **함정**: 기존 `iceberg_maintenance` DAG 의 compaction 대상은 `silver.hotspot_congestion` 뿐. **느린 gold(`fact_hotspot_congestion_5min`)는 미포함** (docstring: gold/dim_place 는 "P1B 후 활성화 의무" 로 미뤄둠). 따라서 gold compaction task 신규 작성이 필요.

## 3. Issue 3 — FastAPI singleton DuckDB 연결 + pyiceberg catalog 의 thread-safety

### 증상

페이지가 `/api/v1/hotspots` 와 `/api/v1/chill-open-now` 를 **병렬(Promise.all)** 로 호출하면 둘 다 즉시(0.06s) 502 실패. 순차 호출에서도 첫 요청 후 간헐 500/502.

### 원인

`api/deps.py` 의 `duck_connection()` / `_catalog()` 가 `@lru_cache` 싱글톤. FastAPI 의 동기(`def`) 엔드포인트는 anyio threadpool 에서 실행되므로, 여러 요청이 **하나의 DuckDB 연결 객체 + 하나의 pyiceberg Catalog** 를 동시에 공유한다. DuckDB connection 과 pyiceberg Catalog 는 스레드 동시 접근에 안전하지 않아 경합/상태 손상이 발생.

(Issue 1 의 500 과 겹쳐 보였으나, Issue 1 = DNS, Issue 3 = thread-safety 로 별개 뿌리.)

### 해결 방향 (Task 11.1 이월)

- 요청마다 DuckDB `con.cursor()` 사용(DuckDB 가 권장하는 동시성 모델), 또는 연결/카탈로그를 요청 스코프로 분리.
- 또는 serving 을 live-read 가 아닌 **사전 적재(Postgres/D1) + 단순 select** 로 바꾸면 동시성·latency 문제가 한꺼번에 사라짐 (§4 참조).

## 4. 아키텍처 인사이트 — Option 5(FastAPI 직결)는 compaction 선행이 전제

Task 11.0 에서 chill_open_now 데이터 source 를 4옵션 trade-off 후 Option 2(Postgres sync)로 결정했다가, Day 8 기존 FastAPI(`/api/chill-open`) 발견으로 **Option 5(FastAPI 직결)** 로 pivot 했다. 본 정찰이 그 결정의 숨은 전제를 드러냈다:

- Option 5 의 "live mart read" 는 **compaction 이 선행돼야** 빠르다. 파일이 9천 개면 첫 요청부터 5-29s 라 Edge cache(60s)로도 못 가린다 (cache 는 첫 요청 비용을 못 줄임).
- 즉 Option 5 vs Option 2 의 진짜 축은 "compaction 했는가" 였다. compaction 후엔 Option 5 가 충분히 빠르고, traffic 이 커지면 그때 Option 2 serving cache 를 추가하는 순서가 자연스럽다.
- **순서 결론**: compaction first → (필요 시) serving cache. compaction 은 어차피 해야 하는 storage 위생 작업이며, 1번 포트폴리오의 "Compaction 도입 예정" 미해결 이슈 closure 서사(CLAUDE.md §10)와 직결된다.
