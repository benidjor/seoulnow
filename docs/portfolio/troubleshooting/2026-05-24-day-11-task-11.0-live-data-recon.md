## 2. Issue 1 — pyiceberg S3 의 `minio` hostname 간헐 DNS 해석 실패

### 증상

로컬 FastAPI `/api/hotspots` 를 반복 호출하면 첫 요청만 200 이고 이후로는 전부 500 이 떨어졌다.

```
hotspots #1 :  5.86s   HTTP 200   ✅
hotspots #2 :  5.50s   HTTP 500   ❌
hotspots #3 :  1.49s   HTTP 500   ❌
```

`/health` 는 언제나 정상이라 프로세스 자체는 살아 있었다. uvicorn 의 stdout 버퍼링 탓에 처음엔 traceback 이 보이지 않았는데, `PYTHONUNBUFFERED=1` 로 재기동하고 포트를 붙잡고 있던 좀비 프로세스를 정리한 뒤에야 실제 에러를 잡을 수 있었다.

```
OSError: When reading information for key 'warehouse/.../snap-...avro'
in bucket 'seoul-warehouse':
AWS Error NETWORK_CONNECTION during HeadObject operation:
curlCode: 6, Couldn't resolve host name      ← 핵심
```

### 원인

`.env` 가 `MINIO_ENDPOINT=http://minio:9000` (docker-internal endpoint, Day 1 환경 편차)으로 잡혀 있다. docker 안에서는 `minio` 가 compose DNS 로 풀리지만, 호스트에서 띄운 FastAPI 는 `/etc/hosts` 의 alias 에 의존한다.

```
.env: MINIO_ENDPOINT=http://minio:9000

  docker 내부 ─── compose DNS ────────▶ minio  ✅ 항상 OK
  호스트 FastAPI ── /etc/hosts alias ──▶ minio  ⚠️ 간헐 실패
```

**DuckDB 쪽은 잘 됐는데 pyiceberg 쪽이 실패했다는 점이 단서였다.** DuckDB(`configure_duckdb`)는 같은 endpoint 를 S3 SECRET 으로 잡아 정상 동작했지만, pyiceberg 의 `plan_files()` 가 쓰는 PyArrow S3FileSystem(AWS C++ SDK)이 manifest(.avro)를 읽을 때 `minio` 해석을 간헐적으로 실패했다. AWS SDK 의 DNS 해석이 `/etc/hosts` 를 일관되게 참조하지 않거나, FastAPI threadpool 스레드에서 negative 캐시 또는 race 가 생긴 것으로 추정된다.

### 해결 (정찰 시점 임시)

endpoint 를 항상 resolve 되는 `localhost` 로 바꿔 띄웠더니 5회 연속 모두 200 이 나왔다. DNS 실패가 endpoint hostname 차이의 증상임이 확정된 것이다.

```bash
MINIO_ENDPOINT=http://localhost:9000 LAKEKEEPER_URL=http://localhost:8181 \
  PYTHONPATH=src uvicorn api.main:app --host 127.0.0.1 --port 8000
```

Task 11.1 에서는 호스트 FastAPI 운영 시 endpoint 를 `localhost`/`127.0.0.1` 로 고정하거나 pyiceberg FileIO 의 S3 설정을 명시 주입해야 한다. 다만 Oracle VM 배포 시에는 compose 네트워크 내부라 `minio` 해석이 정상일 수 있으므로, 환경별로 endpoint 를 분리하는 설정이 필요하다.

## 3. Issue 2 — gold small-file accumulation read latency ⭐ (핵심)

### 증상

Issue 1 을 해결해 500 은 사라졌지만, 이번엔 응답 시간이 5초에서 29초까지 극심하게 출렁였다. 4초로 잡아둔 Edge API timeout 으로는 도저히 버틸 수 없는 수준이다.

```
hotspots #1 :  6.17s    #2 : 28.94s    #3 :  5.05s
hotspots #4 : 26.20s    #5 :  5.02s
```

### 원인 (실측)

단계별로 시간을 쪼개 재보니, manifest 를 읽는 `plan_files()` 와 data 를 읽는 DuckDB `read_parquet` 두 곳에서 시간이 빠졌다.

| 단계 | cold | warm |
|---|---:|---:|
| `build_catalog()` | 0.09s | — |
| `load_table()` | 0.43s | — |
| `scan().plan_files()` — manifest 읽기 | **2.29s** | 0.25s |
| DuckDB `read_parquet` — data 읽기 | **2.78s** | — |

원인은 파일 개수에서 바로 드러났다. **gold 테이블이 총 9,929 row 를 담으면서 data file 을 9,161 개나 쓰고 있었다 — 파일 하나당 약 1 row.** streaming 이 5분마다 micro-batch 로 tiny parquet 를 쌓은 전형적인 small-file 문제다.

```
┌──────────────────────────────────────────────┐
│  data file 개수 :  9,161                       │
│  snapshot 개수  :  3,314                       │
│  총 row 수      :  9,929   →  파일당 약 1 row  │  ⚠️
└──────────────────────────────────────────────┘
```

병목 두 곳(pyiceberg manifest 읽기, DuckDB 파일 read)이 모두 파일 개수에 정비례하므로, 9천 개라는 숫자가 곧 latency 다. 로컬에서는 ~5s 였는데 HTTP 로는 29s 까지 튄 것은, 9천 개 파일을 S3 로 읽는 동안 connection 재시도와 (Issue 3 의) 스레드 경합이 겹쳤기 때문이다.

### 해결 방향 (Task 11.1 이월)

`rewrite_data_files` 로 9,161 개의 작은 파일을 소수의 큰 파일로 합치고 `expire_snapshots` 로 3,314 개 snapshot metadata 를 줄이면, 두 병목 모두 파일 수에 비례하므로 read 가 1초 아래로 떨어질 것으로 본다.

다만 **함정이 하나 있다. 기존 `iceberg_maintenance` DAG 의 compaction 대상은 `silver.hotspot_congestion` 뿐이고, 정작 느린 gold(`fact_hotspot_congestion_5min`)는 빠져 있다**(docstring 상 gold·dim_place 는 "P1B 후 활성화 의무"로 미뤄둔 상태). 따라서 gold 용 compaction task 를 새로 작성해야 한다.

## 4. Issue 3 — FastAPI singleton DuckDB + pyiceberg catalog 의 thread-safety

### 증상

페이지가 두 endpoint 를 `Promise.all` 로 병렬 호출하면 둘 다 0.06초 만에 즉시 502 로 떨어졌다.

```
parallel hotspots :  0.07s   HTTP 502   ❌
parallel chill    :  0.07s   HTTP 502   ❌
```

### 원인

`api/deps.py` 의 `duck_connection()` 과 `_catalog()` 는 `@lru_cache` 로 묶인 싱글톤이다. 그런데 FastAPI 의 동기(`def`) 엔드포인트는 anyio threadpool 에서 실행되므로, 여러 요청이 하나의 DuckDB 연결과 하나의 pyiceberg Catalog 를 동시에 공유하게 된다. 둘 다 스레드 동시 접근에 안전하지 않아 경합과 상태 손상이 일어난다.

```
api/deps.py
   duck_connection()  ──@lru_cache──▶  단일 DuckDB 연결 객체  ┐
   _catalog()         ──@lru_cache──▶  단일 pyiceberg Catalog ┘
                                              │
        FastAPI sync(def) 엔드포인트          │  여러 스레드가
        = anyio threadpool 에서 동시 실행 ────┘  하나의 객체 공유
                                              ▼
                          DuckDB / pyiceberg Catalog 는
                          스레드 동시 접근에 안전하지 않음 → 경합 / 상태 손상
```

Issue 1(DNS)과 증상이 겹쳐 보였지만 뿌리는 다르다. Issue 1 은 DNS, Issue 3 은 thread-safety 다.

### 해결 방향 (Task 11.1 이월)

요청마다 DuckDB `con.cursor()` 를 쓰거나(DuckDB 가 권장하는 동시성 모델) 연결·카탈로그를 요청 스코프로 분리하면 된다. 또는 serving 을 live-read 대신 사전 적재(Postgres/D1) + 단순 select 로 바꾸면 동시성과 latency 문제가 한꺼번에 사라진다(§5 참조).

## 5. 아키텍처 인사이트 — Option 5(FastAPI 직결)는 compaction 이 전제

Task 11.0 에서는 데이터 source 를 네 가지로 비교한 끝에 Option 2(Postgres sync)를 골랐다. 그런데 구현에 들어가면서 Day 8 에 이미 만들어 둔 FastAPI(`/api/chill-open`)를 발견했고, 더 단순한 Option 5(FastAPI 직결)로 방향을 바꿨다. 이번 정찰은 그 Option 5 선택이 사실 "compaction 이 끝나 있다"는 숨은 전제 위에서만 성립한다는 점을 드러냈다.

```
                  compaction 안 함                 compaction 함
                 ┌──────────────┐               ┌──────────────┐
  Option 5       │ live-read     │               │ live-read     │
  (FastAPI 직결) │ 5 ~ 29s  ❌    │   ──그 후──▶   │ < 1s     ✅    │
                 └──────────────┘               └──────────────┘
                        │
                        └─ Edge cache(60s)로도 못 가림 (첫 요청 비용 그대로)
```

파일이 9천 개인 상태에서는 첫 요청부터 5~29초가 걸리고, Edge cache(60s)는 첫 요청 비용을 줄여주지 못한다. **결국 Option 5 와 Option 2 를 가르는 진짜 축은 "compaction 을 했는가"였던 셈이고, 순서는 compaction 을 먼저 하고 그 뒤에도 부담이 남으면 serving cache 를 얹는 쪽이 맞다.** compaction 은 어차피 해야 할 storage 위생 작업이면서, 레시핑의 "Compaction 도입 예정" 미해결 이슈를 닫는 작업(CLAUDE.md §10)과도 곧장 이어진다.

## 7. Task 11.1-A 실행 결과 (2026-05-24)

설계 `docs/superpowers/specs/2026-05-24-day-11-task-11.1a-gold-compaction-realdata-design.md`
대로 gold compaction + expire 를 라이브 스택에 적용하고 latency 를 재측정했다.

### compaction 효과

| 테이블 | files (before → after) | snapshots (before → after) |
|---|---|---|
| `gold.fact_hotspot_congestion_5min` | **9,443 → 3** (직후, 100% 감소) | **3,421 → 5** |
| `silver.hotspot_congestion` | 69 (이미 daily rewrite 중) | **3,429 → 6** |

silver 는 daily rewrite 로 file 은 적었지만 expire 가 비활성이라 snapshot 이 3,429 까지
쌓여 있었다 — `expire_snapshots` 활성화로 함께 정리됐다.

### latency 재측정 (`/api/hotspots`, 단일 요청)

| | before (정찰 §3) | after (compaction 후) |
|---|---|---|
| cold (첫 요청) | 11 ~ 29s | **2.1s** |
| warm | (측정 불가, 매번 느림) | **~35ms** |

`/api/chill-open` 도 49ms. cold 첫 요청 2.1s 도 Edge API 4s timeout 을 하회하고, warm 은
~35ms 라 **Option 5 live-read 가 compaction 전제 위에서 성립함을 실측 확정**했다. 정찰 §5 의
"compaction first → 필요시 serving cache" 중 **serving cache 는 불필요**로 결론.

**동시성 (Issue 3 + compaction 결합)**: `/api/hotspots`+`/api/chill-open` 16 병렬 요청이
compaction 전에는 9천 small-file 탓에 22.58s 걸렸으나(502 는 thread-safety 수정으로 이미 0),
compaction 후 **5.73s, 전부 200** 으로 떨어졌다. thread-safety(cursor + thread-local catalog)
가 502 를 없애고 compaction 이 latency 를 줄이는 두 수정이 결합해 동시 부하에서도 견딘다.
(DuckDB SECRET 이 cursor 로 정상 전파됨을 확인 → cursor 별 재주입/connection pool fallback 불필요.)

### small-file 는 일회성이 아니라 연속 문제

측정 중 gold 가 compaction 직후 3 files 에서 ~10분 만에 9 files 로 늘었다. 원인은
**streaming 이 호스트 프로세스로 가동 중**(`flink_jobs.silver_to_gold` + `producers.hotspot_producer`,
5분 polling)이라 micro-batch 마다 tiny parquet 가 계속 쌓이기 때문이다 (snapshot 시각
14:24 / 14:29 / 14:34 로 5분 간격 확인). 즉 small-file 는 한 번 치우고 끝나는 문제가 아니라
**야간 `iceberg_maintenance` DAG 가 매일 정기적으로 compaction 해야 하는 연속 문제**다 — Task 5
가 gold 를 그 DAG 에 편입한 이유. compaction 은 streaming 동시 쓰기 중에도 Iceberg optimistic
concurrency 로 성공했다.

레시핑의 "Compaction 도입 예정" 미해결 이슈(CLAUDE.md §10)를 gold 에서 닫음.
