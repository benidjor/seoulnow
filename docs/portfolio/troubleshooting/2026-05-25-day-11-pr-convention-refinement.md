## 0. 요약

Day 10 에 이어 Day 11 에서도 PR 본문 컨벤션 회귀 발생. 근본 원인은 실제 템플릿 파일 미확인 + 작성 직전 grep self-check 누락. 정정 과정에서 구조/문체 규칙 4건을 정착시키고 PR #76 으로 템플릿에 명문화.

| 회귀 | 정정 |
|---|---|
| 템플릿 구조 미준수 (임의 §1 구성, §5/§6 누락) | 실제 템플릿 7-섹션 복원 |
| 금지 표현 1건 (프로젝트 순번 지칭) | 프로젝트명(레시핑) 직접 표기 |

## 1. 템플릿 미준수 — 메모리 요약만 보고 실제 파일 미확인

### 1.1. 발생 과정

PR #74/#75 본문 v1 을 `korean-conventions` 메모리의 구조 요약만 참고해 작성. 실제 `.github/PULL_REQUEST_TEMPLATE.md` 를 열지 않음.

- §1 개요 하위를 배경 / 목표 / 범위로 구성(실제 템플릿은 배경 / 목적 / 결과 / 기대 효과 4개).
- §5 장애 시나리오 & 롤백, §6 체크리스트 통째 누락.
- 섹션 번호에 점 미표기(`### 1.1` vs 템플릿 `### 1.1.`).

### 1.2. 식별

사용자 지적 — "커밋, PR 템플릿을 확인하고 작성한 것이 맞아?". `.github/PULL_REQUEST_TEMPLATE.md` 정독 후 4,265바이트 템플릿이 7-섹션 + §1 하위 4개 + §6 체크리스트 6항목을 명시함을 확인.

### 1.3. 정정

두 PR 본문을 템플릿 7-섹션으로 전면 재작성. 메모리에는 이미 "PR 작성 시 실제 템플릿 파일 확인 의무" 가 있었으나 적용 실패 — 메모리 갭이 아닌 적용 실패.

## 2. 톤 위반 — grep self-check 누락

### 2.1. 발생 과정

`gh pr create` 직전 금지어 grep 을 돌리지 않음. PR #74 본문에 프로젝트를 이름(레시핑) 대신 순번으로 지칭하는 금지 표현 1건 + `closure` 잔류.

### 2.2. 정정

프로젝트명 직접 표기로 교체 + `closure` 풀어쓰기. 동일 표현이 커밋된 정찰 doc §5(Day 11.0 시점 작성)에도 있어 함께 정정. `korean-conventions` 의 "`gh pr edit` 직전 매번 grep 의무" 가 명시돼 있었음에도 누락 — Day 10 PR #57 회귀와 동일 패턴.

## 3. 구조 / 문체 규칙 4건 정착

사용자 피드백을 반영해 다회 재작성하며 규칙 4건 확정.

### 3.1. 다중 그룹 섹션은 `### N.M.` 헤더 승격

§1 은 `###` 하위 헤더인데 §2 이후가 평면 불릿이라 시각 비대칭 발생.

- 명명 그룹 2개 이상인 섹션(§2 의사결정 / §3 변경 / §5 롤백)은 `### N.M.` 헤더로 그룹화.
- 단일 리스트 섹션(§4 검증 / §6 체크리스트 / §7 레퍼런스)은 평면 유지(과잉 구조 회피).
- 효과: §1 과 시각 정렬 + GitHub PR 아웃라인(TOC)에 그룹 노출(`###` 만 잡힘, bold 불릿은 미노출).

### 3.2. §3 변경 사항 = 논리 그룹 → 파일 하위 불릿

파일 평면 나열 대신 논리 변경 그룹으로 묶고 그 결과 바뀐 파일을 하위 불릿으로.

- 예: `### 3.1. gold compaction 복원` → 한 문장 설명 → `compaction_gold.py` / `iceberg_maintenance.py` 불릿.

### 3.3. 헤더 직후 바로 불릿 금지

모든 헤더(`##` / `###` / `####`) 직후 한두 문장 내용을 먼저 쓰고 불릿은 그 다음.

- 불릿 앞 내용 문장이 여러 개면 마침표(.) 기준 줄바꿈(한 문장 = 한 줄).

### 3.4. 서술형 종결어미 회피, 명사형 우선

`~다` / `~했다` / `~된다` 회피, 명사형(`~함` / `~음` / 명사 종결) 우선.

- PR #73 의 "전면 명사형 stiff" 우려는 연결 문단 통째 명사형화 한정 — bullet / 짧은 lead 는 명사형이 자연.
- 격식 종결(`~습니다`)은 금지 유지.

## 4. PR #76 으로 템플릿 명문화

규칙 4건을 `.github/PULL_REQUEST_TEMPLATE.md` 상단 주석 + §3 골격에 박음. 메모리 SoT 외에 레포 템플릿에도 박아 다음 PR 부터 일관 적용.

## 5. 학습

Day 10·11 연속 회귀에서 뽑은 재발 방지 원칙.

- PR 본문 작성 전 `.github/PULL_REQUEST_TEMPLATE.md` **실제 파일** 정독 의무. 메모리 요약은 보조.
- `gh pr create` / `gh pr edit --body-file` 직전 grep self-check 절대 의무 — 금지어 + jargon + tilde-range + 헤더-직후-불릿.
- 헤더-직후-불릿 점검 명령: `awk '/^#{2,4} /{h=NR;next} h && NR==h+2 && /^[-|]/{print NR}{if(NR>h+2)h=0}' <body>`.
- Day 10·11 연속 회귀 = 작업이 길고 병렬(3-트랙 + 다중 PR)일 때 작성 직전 점검이 누락되는 패턴. 체크리스트화로 회피.
