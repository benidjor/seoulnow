## 2. plan 4 buggy patterns 사전 식별 (Task 4.1)

silver_to_gold.py 작성 전 plan L2212~L2338 코드를 main HEAD `bronze_to_silver.py` 와 비교 → 4건 incompatibility 식별. silver fix 의 학습 위에서 사전 catch.

### 2.1. 4-part table name — Flink SQL parser 한정

plan 의 코드:

```python
cat = warehouse_namespace()  # → "seoul"
t_env.execute_sql(f"""
    INSERT INTO ice.{cat}.gold.fact_hotspot_congestion_5min
    SELECT ... FROM ice.{cat}.silver.hotspot_congestion ...
""")
```

`ice.seoul.silver.hotspot_congestion` 는 4 부분. Flink SQL parser 는 `catalog.database.table` 의 3 부분만 받음. `lib/iceberg_sink.py:register_iceberg_catalog` 는 이미 `ice.bronze` / `ice.silver` / `ice.gold` 의 flat database 로 등록되어 있어 4-part 가 redundant.

**fix**: 모든 SQL 에서 `ice.silver.hotspot_congestion` / `ice.gold.fact_hotspot_congestion_5min` 의 3-part 사용. `warehouse_namespace()` 호출 자체 제거.

### 2.2. ClassLoader fix 누락 — silent commit fail 재발 risk

plan 의 `build_env()` 가 `pipeline.jars` / `parallelism.default` / `execution.checkpointing.interval` 만 set. Day 4 Task 1 silver fix 의 핵심 1줄 `classloader.parent-first-patterns.additional` 누락.

미적용 시 IcebergStreamWriter.prepareSnapshotPreBarrier 단계에서 `com.codahale.metrics.Histogram` LinkageError 매번 fail → silver→gold sink 도 같은 silent commit fail 재발.

```python

t_env.get_config().set("table.dynamic-table-options.enabled", "true")
t_env.get_config().set("parallelism.default", "1")
t_env.get_config().set("execution.checkpointing.interval", "30 s")
```

### 2.4. watermark 누락 — TUMBLE 윈도우 close 안 됨

plan 의 source view:

```sql
CREATE TEMPORARY VIEW silver_stream AS
SELECT *,
  CAST(api_response_ts AS TIMESTAMP_LTZ(3)) AS event_time
FROM ice.silver.hotspot_congestion
```

`event_time` 만 만들고 `WATERMARK FOR event_time AS ...` 절 없음. TUMBLE 윈도우는 watermark 가 없으면 close 되지 않아 INSERT 가 영원히 sink 로 안 흘러감 → Gold row 0 건.

## 3. silver_stream_wm watermark 우회 (Task 4.1)

§2.4 의 watermark 부여를 어디에 어떻게 박을지 결정 과정.

### 3.1. 시도 1 — CREATE TEMPORARY VIEW 에 WATERMARK 절

```sql
CREATE TEMPORARY VIEW silver_stream AS
SELECT
  *,
  CAST(api_response_ts AS TIMESTAMP_LTZ(3)) AS event_time,
  WATERMARK FOR event_time AS event_time - INTERVAL '1' MINUTE
FROM ice.silver.hotspot_congestion
```

→ Flink SQL parser 거부. CREATE TEMPORARY VIEW 는 SELECT 절만 받고 WATERMARK 절 미수용.

### 3.2. 시도 2 — Iceberg silver catalog table 자체에 WATERMARK 부여

`ice.silver.hotspot_congestion` 테이블 정의 자체에 watermark 추가:

```sql
ALTER TABLE ice.silver.hotspot_congestion
ADD WATERMARK FOR api_response_ts AS api_response_ts - INTERVAL '1' MINUTE
```

→ Iceberg catalog table 은 schema/partition 만 정의 가능. Flink-only WATERMARK 는 catalog 가 받지 못함. 게다가 silver 는 `bronze_to_silver.py` 의 INSERT sink 이기도 해서 watermark column 추가 시 schema mismatch.

### 3.3. 결정 — direct iceberg connector 옵션의 임시 source-table

같은 Lakekeeper REST + S3FileIO 옵션을 catalog 와 별도 direct connector (`'connector' = 'iceberg'`) 옵션으로 다시 박은 임시 source-table `silver_stream_wm` 생성. WATERMARK 절은 이 임시 source-table 의 schema 정의 단계에서 가능.

```sql
CREATE TEMPORARY TABLE silver_stream_wm (
  area_code STRING,
  district STRING,
  gu_code STRING,
  congest_level_score INT,
  population_min INT,
  population_max INT,
  api_response_ts TIMESTAMP(3),
  event_time AS api_response_ts,
  WATERMARK FOR event_time AS event_time - INTERVAL '1' MINUTE
) WITH (
  'connector' = 'iceberg',
  'catalog-name' = 'silver_src_wm',
  'catalog-type' = 'rest',
  'uri' = '...',
  'warehouse' = 'seoul',
  'catalog-database' = 'silver',
  'catalog-table' = 'hotspot_congestion',
  'io-impl' = '...',
  's3.endpoint' = '...',
  ...
  'streaming' = 'true',
  'monitor-interval' = '30s'
)
```

물리 데이터는 catalog table `ice.silver.hotspot_congestion` 과 동일 (같은 Lakekeeper warehouse 의 같은 table 을 가리킴). `'catalog-name' = 'silver_src_wm'` 은 connector 의 필수 옵션 (처음 누락 시 `NullPointerException: Table property 'catalog-name' cannot be null` 발생).

### 3.4. drift 안전망

`silver_stream_wm` 의 7 컬럼 hardcode (silver 17 컬럼 중 SELECT/GROUP BY 가 참조하는 컬럼만). silver schema 변경 시 silently miss-projection 가능.

mitigation — `create_silver_source_with_watermark` docstring 에 명시:

```

컬럼) 가 변경되면 본 함수의 7 컬럼 (area_code / district / gu_code /
congest_level_score / population_min / population_max / api_response_ts)
도 동기 수정 필요.
```

## 4. DuckDB iceberg ext + Lakekeeper UUID path 호환 안 됨 (Task 4.2 / 4.3)

plan 의 DuckDB `iceberg_scan('s3://seoul-warehouse/warehouse/<layer>/<table>')` 가 Task 4.1 verification 시점에 실패. 두 단계의 우회 시도 끝에 정공법 도입.

### 4.1. iceberg_scan(s3://...) 실패 진단

Task 4.1 의 Step 3 verification (plan L2322~L2330) 그대로 실행:

```python
con.execute("INSTALL iceberg; LOAD iceberg;")
con.execute("CREATE OR REPLACE SECRET (...)")
con.execute("SELECT * FROM iceberg_scan('s3://seoul-warehouse/warehouse/gold/fact_hotspot_congestion_5min') LIMIT 10").fetchall()
# IOError: No such file or directory
```

→ Lakekeeper REST 가 vend 하는 actual path 가 디렉토리 이름이 아닌 UUID-prefix.

```bash
mc ls minio/seoul-warehouse/warehouse/
# 01928c0a-xxxx-xxxx-xxxx-xxxxxxxxxxxx/   <- warehouse UUID
#   01928c0b-yyyy-...../                  <- table UUID
#     metadata/
#     data/
```

iceberg_scan 은 path 안의 metadata.json 을 자동 lookup 하지만 prefix 가 UUID 라 디렉토리 이름 path 로는 도달 못함. **Lakekeeper REST + UUID path 환경에서는 iceberg_scan 직접 호출 불가.**

### 4.2. pyiceberg to_arrow() + pyarrow 11 호환 충돌

차선책으로 pyiceberg `t.scan().to_arrow()` 시도:

```python
t = catalog.load_table("gold.fact_hotspot_congestion_5min")
df = t.scan().to_arrow().to_pandas()
# TypeError: concat_tables() got an unexpected keyword argument 'promote_options'
#   at pyiceberg/io/pyarrow.py: scan_to_arrow
```

pyarrow `concat_tables` 의 `promote_options` 인자는 pyarrow 14+ 에서 추가. 본 환경은 PyFlink 1.20 이 pyarrow 11 에 transitive dependency 로 묶여 있어 upgrade 불가.

`uv pip list | grep pyarrow` 결과 pyarrow 11.0.0. PyFlink 1.20 의 release notes 에 명시된 호환 매트릭스도 11.0.

### 4.3. 결정 — pyiceberg plan_files() + DuckDB read_parquet

책임 분리 우회:
- **pyiceberg = catalog lookup 만**: `table.scan().plan_files()` 로 실제 parquet path 리스트 받음. UUID path resolve 됨
- **DuckDB = parquet decode 만**: `read_parquet(?, hive_partitioning=true)` 로 path 리스트 직접 read

```python
table = catalog.load_table("gold.fact_hotspot_congestion_5min")
file_paths = [f.file.file_path for f in table.scan().plan_files()]
# ['s3://seoul-warehouse/warehouse/<uuid>/<uuid>/data/district=강남구/00000-...parquet', ...]

con = duckdb.connect()
con.execute("INSTALL httpfs; LOAD httpfs;")
con.execute("CREATE OR REPLACE SECRET (TYPE S3, KEY_ID, SECRET, ENDPOINT, URL_STYLE 'path', USE_SSL false, REGION)")
con.execute("SELECT ... FROM read_parquet(?, hive_partitioning = true) WHERE ...", [file_paths])
```

장점:
- pyarrow 11 / 14 무관 (pyiceberg 의 plan_files 만 사용, to_arrow 미사용)
- Lakekeeper UUID path 정상 (pyiceberg 가 catalog metadata 에서 path resolve)
- hive_partitioning=true 가 Iceberg 의 Hive-style 파티션 (e.g. `district=강남구`) 을 column 으로 자동 복원

`slo_metrics.fetch_samples_from_gold` (Task 4.2) 와 `duckdb_check._configure_duckdb` + `_table_paths` (Task 4.3) 양쪽에 동일 패턴 적용. Day 5 진입 전 `flink_jobs/lib/duckdb_iceberg.py` 추출 예정.

## 5. SLO percentile p95 414 → 393 정정 (Task 4.2)

plan L2381 의 expected 가 fixture / 정의와 안 맞음 식별.

### 5.1. fixture 와 정의

```python
samples = [60, 90, 120, 150, 180, 210, 240, 300, 360, 420]   # n=10
# 정의 (plan L2446~L2456): linear interpolation, rank = p * (n - 1)
```

### 5.2. p95 직접 계산

`rank = 0.95 * 9 = 8.55` → `samples[8]=360`, `samples[9]=420`, `frac=0.55` → `360 + 60 * 0.55 = 393`

cross-check:

```python
import numpy as np
np.percentile([60, 90, 120, 150, 180, 210, 240, 300, 360, 420], 95, method='linear')
# 392.99999999999994 → round → 393
```

plan 의 414 와 안 맞음.

### 5.3. 414 의 정체 — p99 truncation

같은 fixture 의 p99:

`rank = 0.99 * 9 = 8.91` → `360 + 60 * 0.91 = 414.6` → truncate → **414**

→ plan author 가 p99 truncation 값을 p95 칸에 잘못 넣은 hand-calc 오류로 판단.

### 5.4. 결정 — 393 정정 + int(round) 채택

- test expected 393 으로 정정 (`tests/unit/test_slo_metrics.py:test_summarize_returns_p50_p95_p99_max`)
- docstring 으로 정정 사유 명문화 — 같은 실수 reader 재발 방지
- `_percentile` 구현은 `int(round(...))` 채택. plan 원본의 `int(...)` truncation 보다 부동소수점 epsilon 안정. numpy `method='linear'` 와 동일 결과

## 6. duckdb_check 4번 query 명세 위반 (Task 4.3 spec review)

implementer 가 작성한 4번째 query (`scripts/duckdb_check.py:132-142`):

```sql
SELECT district, window_start, avg_congest_score
FROM read_parquet(?, hive_partitioning = true)
QUALIFY row_number() OVER (PARTITION BY district ORDER BY window_start DESC) = 1
ORDER BY district
```

plan L2602~L2606 의 명세:

```sql
SELECT district, avg_congest_score
FROM iceberg_scan(...)
QUALIFY row_number() OVER (PARTITION BY district ORDER BY window_start DESC) = 1
ORDER BY avg_congest_score DESC
```

차이:
- `window_start` 컬럼 implementer 추가 — plan 은 2 컬럼 명세
- `ORDER BY district` (alphabet) — plan 은 `avg_congest_score DESC` 명시

smoke run 시점은 모든 district 의 avg_congest_score=1.0 동률이라 정렬 오류 가시화 안 됨. 분포 발생 시 잘못된 순서로 표시.

spec reviewer subagent 가 이 deviation 을 실제 출력 + plan 명세 비교로 catch. 2-line fix (`97214c3`):

```diff
-                SELECT district, window_start, avg_congest_score
+                SELECT district, avg_congest_score
                 FROM read_parquet(?, hive_partitioning = true)
                 QUALIFY row_number() OVER (
                     PARTITION BY district ORDER BY window_start DESC
                 ) = 1
-                ORDER BY district
+                ORDER BY avg_congest_score DESC
```

ruff + 재실행 verification 통과.

## 8. 학습 패턴

### 8.1. plan deviation 식별 → 적용 → docstring + commit body 양쪽 명문화

### 8.2. DuckDB iceberg ext vs pyiceberg + read_parquet

Lakekeeper REST + UUID path 환경에서는 `iceberg_scan(s3://...)` 직접 호출이 정공법 아님. pyiceberg `plan_files()` + DuckDB `read_parquet(hive_partitioning=true)` 가 정공. dbt 모델 / 화면 spike 진입 시점에도 같은 패턴 재사용 가능.

`hive_partitioning=true` 가 Iceberg 의 Hive-style 파티션을 컬럼으로 자동 복원하는 게 결정적. partition 컬럼 (`district`) 이 SELECT 시 정상 노출.

### 8.3. spec / plan 의 hand-calc 수치도 검증 필요

414 / 393 사례 (§5). plan 의 expected 가 정의 (linear interpolation) 와 fixture 자체 계산 결과를 일치시키지 않은 경우. **TDD 의 Red 단계에서 expected 값을 numpy / 수동 계산으로 cross-check 하는 절차** 가 plan 위 작업의 표준이 되어야 함. 다른 percentile / aggregate / window 경계 수치도 Day 5+ TDD 시 동일 cross-check 필요.

### 8.4. TDD 영역 분리 — 순수 함수 vs streaming/I/O

Task 4.2 의 `compute_freshness_seconds` / `_percentile` / `summarize` = 순수 함수, unit test 5건. `fetch_samples_from_gold` = I/O, smoke run 검증.
Task 4.1 의 `silver_to_gold` = streaming job, smoke run 30분 검증.
Task 4.3 의 `duckdb_check` = I/O 스크립트, 실행 검증.

### 8.6. spec reviewer subagent 의 catch 가치
